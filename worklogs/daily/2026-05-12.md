## Approach

1. Build a complete source pool before sampling.
2. Assign every candidate row to a single accounting bucket so the evaluation set is mutually exclusive and collectively exhaustive.
3. Separate reporting populations:
   - balanced evaluation set for per-code quality;
   - production holdout for production-weighted monitoring;
   - fixture set for deterministic system or postprocess behavior;
   - audit set for ambiguous or internally inconsistent labels.
4. Require version and configuration alignment before rerun.
5. Use synthetic or generated conversations only for coverage gaps that production cannot fill.

## Data Selection And Alignment

The selection gate is now defined before running additional evaluation:

- Start from the accounted production source pool.
- Keep only rows with the required rerun inputs for evaluation use.
- Confirm the row was produced under a known configuration version.
- Prefer rows whose prompt and call-code definitions match the current approved configuration.
- Route older or mismatched configuration groups to historical recovery or separate reporting, not the main current-config evaluation set.
- Route internally inconsistent labels to Auto-QC/audit instead of treating them as high-confidence labels.

This prevents mixing rows from different classifier versions and makes the future evaluation result easier to defend.
