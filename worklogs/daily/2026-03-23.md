## AGI-6751 · Master Call Code

**Why.** The Master Call Code reference page used "customer" and "call receiver" interchangeably across the Short Description and Explanation columns. For compliance-class codes (DNC, DIS, ATY, FRA), the distinction is regulatory — **pre-IDV** (phone-number-level, applies to *anyone* who picks up that number) signals must not be classified as **post-IDV** (account-level, applies only after verified ID). Mis-classifying a phone-number-level signal as account-level is a 30-day-suppression compliance risk: the dialer keeps calling the same number because the account-level flag isn't honored at dial-time.

> ### 🛠 What I did
> Posted the explicit question in JIRA — *"are compliance call codes applicable for 'customer' or any call receiver?"* — and pointed at the column inconsistency plus the downstream compliance risk if pre-IDV signals get treated as account-level. *(JIRA comment, 2026-03-23 09:58 +0700)*

**Result.** Same-day Huong Bui defined the canonical split — phone-number level (DNC, CDS) pre-IDV, account level (DIS, ATY) post-IDV, FRA requires IDV — and migrated the canonical reference off the Confluence page onto a Google Sheet with an explicit `Applies To` column. The boundary pinned here flows into AGI-9466 AC routing and the `call_code_descriptions` payload that ships with every postcall classifier prompt.

**Credits.** Huong Bui (AGI-Product) — defined the canonical pre/post-IDV split; owns the new master-sheet reference.

**Links.**
- AGI-9466 — Group-1 codes (NNP/DIM/GVT) inherit this pre/post-IDV boundary.
- `call_code_descriptions` (live postcall prompt payload) — the structure decisions made here propagate through every downstream classifier run.
- Master Call Code Google Sheet (replaces the legacy Confluence page; carries the `Applies To` column).

## AGI-7440 · FTP coding for technical-failure calls

**Why.** A flagged call where the bot went silent immediately after switching to DTMF; `payment_result` never returned from the payment API. Product asked whether `FTP` (Fail to Pay) was the right disposition or if the call should be coded differently because the consumer had completed their side of the flow correctly.

> ### 🛠 What I did
> Explained the technical mechanism in JIRA — without `payment_result`, the call effectively fails to process to payment from an operations standpoint, so `FTP` is the best operational descriptor for this class. Invited Product to propose alternatives if a better code existed for similar technical-fail-then-DTMF-retry cases. *(JIRA comment, 2026-03-23 15:30 +0700)*

**Result.** Product validation aligned on `FTP` as the operational descriptor for this class. Three weeks later (Apr 10), Thanh Vo shipped the postprocess condition fix that handles the precise edge case raised here — the same class of call now classifies as `PPA` instead of `FTP` when `payment_init` fired but `payment_result` was missing. AGI-7440 closed Apr 10.

**Credits.** Thanh Vo (DS) — shipped the FTP → PPA postprocess condition fix. Lien Nguyen (UAT) — confirmed the fix.

**Links.**
- Postcall postprocess conditions — the rule layer that re-ranks LLM output based on state-machine signals like `payment_init` / `payment_result`.
- AGI-9466 — the FTP/PPA distinction informs operational-class code definitions for the Affirm bot.
