## Clean-data audit pipeline v1 + codebase-search tooling

**Why.** Tue's `tc_clean_column_airflow_map_v1.csv` was hand-cut against CSV2 notebooks. That doesn't scale: `mbf_cs_airflow_202208` has four dev trees (CSV2 / CSV3 / CSV4 / ISV1) with dozens of notebooks each, and many `f_name` lookups (e.g. `rad.run_prepare_account_deactivation_at_upto_date`) resolve to multiple candidate `pipeline/*/base.py` files with different bodies. Need a programmatic auditor that walks notebooks → extracts Airflow IDs → resolves the Spark-submit `f_name` → parses the `read.parquet → selectExpr/where/dropDuplicates` chain to land on column usage. Plus a codebase-search tool that disambiguates which `base.py` actually owns a given `f_name`, since a flat `grep` across the four trees gives ambiguous hits.

> ### 🛠 What I did
>
> - **Built `scripts/build_clean_data_audit.py`** — the audit pipeline. Walks notebooks under `csv*_dev/lib_notebook_examples/`, extracts `<component>/<name>` Airflow IDs from raw cells (using Tue's `extract_notebook_airflow_ids.py` pattern), then resolves the Spark-submit `f_name` to the correct `pipeline/<f_name_root>/base.py` and parses out the columns actually read. Output schema mirrors Tue's hand-cut: `tc_file_name | clean data | hadoop path | TS schema | Airflow ID | f_name | columns`.
> - **Materialized `data/etl/sysconf_clean_paths_inventory.tsv`** — every `data_tc_*` / `idata_*` / `feat_pop_*` key in `common_dev/lib/sysconf/base.py` flattened to a TSV, so the pipeline trace can resolve every `conf[...]` lookup mechanically without re-parsing the sysconf module per notebook.
> - **Shipped the first end-to-end audit run** → **`clean_data_audit_20260325.csv`** (65 KB) + **`clean_data_audit_COVERAGE_SUMMARY.txt`**. Same primary-key shape as Tue's hand-cut v1, but generated by code from the notebooks rather than hand-walked.
> - **Stood up codebase-search tooling**: `src/search_mbf_text.py` — plain-text grep with notebook-cell awareness (skips `outputs` / `metadata`, parses `source` arrays); `src/search_mbf_codebase.py` — symbol-aware search across the four dev trees so an `f_name` like `rad.run_prepare_account_deactivation_at_upto_date` can be located across all `csv*_dev/lib/pipeline/rad/base.py` candidates and disambiguated by which one the corresponding notebook actually imports.

**Result.** From-zero audit pipeline running by lunch; first end-to-end output by 14:26. The 65 KB CSV + coverage summary become the **objective baseline** that Thu's 5-LLM bake-off (gemini31 / opus46 / gpt54 / sonnet46 / opus46v2) is benchmarked against. Search tooling unblocks the `f_name → base.py` disambiguation that the pipeline depends on for every multi-tree function.
