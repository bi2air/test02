## 5-LLM bake-off + constitution + worker6 consolidation

**Why.** Wed's `clean_data_audit_20260325.csv` was a single-model pass. The audit task has nuance — extracting which `tc_*` columns each `pipeline/*/base.py` function actually reads requires correctly parsing `selectExpr / where / dropDuplicates` chains wrapped around `read.parquet`, plus handling notebook idioms (`<component>` blocks, raw cells with XML, conditional `RUN_WITH_AIRFLOW` branches). Different LLMs hallucinate different parts of this. Running 5 variants in parallel against the same task is the cheapest way to (a) pick the model that handles MBF's monorepo idiom best, (b) catch edge cases one model misses but another catches, and (c) calibrate confidence in the headline numbers before they go to bid sizing.

> ### 🛠 What I did
>
> - **Five parallel audit-build attempts**, each in its own model-named subfolder, all targeting the same task spec:
>   - **`gemini31/`** (Gemini 3.1) — pipeline split across 6 small Python files (`process_audit.py`, `parse_audit.py`, `parse_audit_full.py`, `convert_to_md.py`, `test_ast.py`, `generate_report.py`); AST-based notebook parsing approach. Outputs: `audit_results.csv` + `final_clean_data_report.{csv,md}` + `gap_analysis_report.md`.
>   - **`opus46/`** (Opus 4.6) — tighter design, 2 files (`parse_clean_data_audit.py` + `generate_report.py`); outputs `clean_data_audit_opus_0326.csv` + `clean_data_report.md` + `gap_analysis.md`.
>   - **`gpt54/`** (GPT-5.4) — only entry that shipped with `test_build_clean_data_audit.py` and a `README.md`; most disciplined repo layout. Outputs: `clean_data_audit.csv` + `clean_data_report.md` + `gap_analysis.md`.
>   - **`sonnet46/`** (Sonnet 4.6) — single-file `build_audit.py` → `audit.csv` + `report.md`; simplest output, no gap analysis (intentional minimum-viable variant).
>   - **`opus46v2/`** (Opus 4.6, second pass with refined task spec post-`consitution.md`) — `build_audit.py` + `audit.csv` + **`coverage_summary.txt`** + `gap_analysis.md` + `report.md`. Adds the coverage_summary that the first Opus run was missing.
> - **Drafted `consitution.md` mid-day**, then **`consitution_v2.md`** in the late afternoon — the tightened task-spec / governance doc the bake-off taught me to write. v1 codifies the column-trace contract (anchors, what counts as a column read, what gets reported in the gap section). v2 is ~3× the size: adds the rules surfaced by the bake-off (handle conditional `RUN_WITH_AIRFLOW` branches; treat notebook header IDs like `100-000` as not-task-IDs; the `<prepend>` chain for ordering).
> - **Synthesized `evaluation_report.md`** — head-to-head evaluation of the 5 outputs against the constitution: which model captured which `tc_*` table best, which hallucinated columns that aren't in `selectExpr`, which under-reported. Output drives the worker6 design.
> - **Wrote `reflection.md`** — methodological notes on what the bake-off taught about the task itself (column-trace edge cases, where the LLMs converge vs diverge), separate from picking a winner.
> - **Started `worker6_v2`** in the evening — first attempt at the consolidated production auditor incorporating the bake-off lessons. Then **`worker6_v2_nb`** ~3 hours later as a notebook-flavored variant. Both ship `build_audit.py` + `audit.csv` + `gap_analysis.md` + `report.md`. v3 → v4 final ship the next morning.
> - **Synced `PLAYBOOK_idata_clean_column_trace_v1.md` into the monorepo** — keeping the playbook colocated with the code it navigates, so anyone reading `mbf_cs_airflow_202208/` finds the navigation reference next to the source.

**Result.** Five model outputs head-to-head + an evaluation + a reflection + a v2 task-spec + two evening worker6 iterations, all in one day. Bake-off surfaces model-specific failure modes (which AST-parser chokes on which notebook idiom). `consitution_v2.md` becomes the canonical task spec; **`worker6_v2_nb`** is the bridge to Fri's `worker6_v3 → v4` final ship.
