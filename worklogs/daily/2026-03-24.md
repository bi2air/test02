## Code-level column-trace audit — playbook + v1 artifacts + dictionary v3

**Why.** The Mar 19-20 playbook matched MBF-delivered CSV columns to the TS_Schema spreadsheet. That answers "what columns came in"; it does **not** answer "which columns the production scoring pipeline actually reads." Pipeline reality is in the legacy `mbf_cs_airflow_202208` monorepo (CSV2/CSV3/CSV4/ISV1 dev trees, ~2022-12 vintage) — Airflow-orchestrated notebooks calling Spark jobs in `pipeline/*/base.py`. The audit has to walk **notebook `<component>` raw cells → Airflow ID → producing function → `read.parquet` → `selectExpr/where/dropDuplicates`** to land on real column usage. Without that, the bid sizing (which clean tables MBF must commit to delivering, with which columns) is guesswork.

> ### 🛠 What I did
>
> - **Authored `PLAYBOOK_idata_clean_column_trace_v1.md`** — navigation guide for tracing `idata` jobs back to the exact clean-table columns they consume. Anchors: `common_dev/lib/sysconf/base.py` as config source-of-truth (every `data_tc_*` / `idata_*` / `feat_pop_*` path key); `csv*_dev/lib_notebook_examples/` raw `<component>` cells as the Airflow-ID source (e.g. `<name>100-A01</name>` is the canonical task ID, not the notebook header `100-000`); the `.selectExpr / .where / dropDuplicates` chain immediately after `read.parquet(conf[...])` as the column-truth boundary. Includes the verification rule: open the `run_*_at_upto_date` function and read what it actually selects — never assume `tc_<table>` = its full column list.
> - **Built `tc_clean_column_airflow_map_v1.csv`** (the deliverable artifact, primary key = `tc_file_name`) with format documented in the README: `tc_file_name | clean data | hadoop path | TS schema | Airflow ID | f_name | columns`. First-pass hand-cut walking CSV2 notebooks; populates the columns each Airflow job consumes from each `tc_*` table.
> - **Built `tc_clean_column_vs_fintech_sample_v1.csv`** — cross-walk between what the pipeline expects (per the trace above) and what the encrypted fintech sample (Mar 20 arrival) actually delivers. Surfaces the structural gaps that Apr 8's MHTT audit later re-confirms: raw CDR vs hourly aggregate, OG/IC voice direction-split, missing tables.
> - **Wrote `scripts/extract_notebook_airflow_ids.py`** — pulls `<component>/<name>` from notebook raw cells so the Airflow ID column is mechanical, not hand-curated. Becomes the upstream extractor for Wed's full audit-pipeline build.
> - **Built dictionary v3 pipeline** — `src/build_mbf_dictionary_v3.py` + `src/mbf_dictionary_v3_mapping_overrides.yaml` (human-curated layer on top of the auto-extracted dictionary) → `data/customer/mbf_dictionary_v3.xlsx`. v3 is the upgrade over the canonical `DICTIONARY MBF_2.xlsx` (Mar 20 arrival): rebuilds the dictionary from the legacy code's actual column usage rather than from the MBF-side documentation alone.

**Result.** Three deliverables land that anchor the rest of the week: the PLAYBOOK as the navigation reference, `tc_clean_column_airflow_map_v1.csv` as the hand-cut baseline, `mbf_dictionary_v3.xlsx` as the upgraded MBF-side dictionary. v1 is hand-cut on purpose — Wed automates it (`scripts/build_clean_data_audit.py`) and uses today's hand-cut as the golden output to validate against.
