## DCI-321 · CSV4 scoring pipeline — minimum clean-data weeks audit

Need an authoritative top-down audit walking the DAG backward from `csv4_score_prediction` (component **118-B01**) through `layer2_combined343` (**107-A02**) and `layer1_combined179` (**107-A01**) to the underlying clean feeds, so cloud-cutover sizing has a defensible number.

> ### 🛠 What I did

> - **`csv4_pipeline_dag_map.md` + `.drawio`** — full onion-layer DAG: each node tagged with artifact, producing function, code location, inputs, lookback. Parallel branches use `max(weeks)`, sequential stages on one branch use `sum`.
> - **`layer1_pipeline_dag_map.md` + `layer1_flowchart.md`** — Layer 1 sub-DAG (the inner-most aggregation layer; inherits depth from 7+ feature branches).
> - **`dag_airflow_detail_map.md`** — Airflow operator-level mapping (which DAG/task produces each artifact).
> - **`csv4_clean_data_audit_report.md`** — executive summary with headline numbers and bottleneck call-outs.
>

## Reusable methodology — two complementary reverse-engineering skills authored mid-audit

**Why.** Audits like the CSV4 walk happen recurrently (every cloud cutover, every scoring-platform migration, every "how many weeks of clean data do we need?" question). The morning DAG walk surfaced a methodological insight worth codifying as a reusable skill: **notebooks are the first ground truth for DAG structure** (the `<airflow_upstream>` XML tags in raw cells define edges, before `base.py` ever needs to be read). That insight + the executable date-arithmetic verifier are the difference between "trust me" and "re-derivable in 30 minutes by anyone."

> ### 🛠 What I did
> Distilled the audit methodology into a **two-skill complementary pair** at `.claude/skills/`, finalized at 17:20 alongside the audit report:
> - **`reverse-telco-codebase.md`** (17:20) — canonical skill: code-first, onion-peel from final score. Defines the 5 questions to answer per layer (what reads, at what offset, through which helper).
> - **`reverse-telco-codebase-reference.md`** (12:03) — codebase-specific entry points, conf-key mappings, verified DAG skeleton, known discrepancies (this codebase's `ghelper.py` quirks).
> - **`verify_date_arithmetic.py`** (12:04) — executable Python script replicating `ghelper.py` date functions; verifies all 3 deep chains. Run as `python3 verify_date_arithmetic.py --upto YYYYMMDD`. **The chained-week math is what the audit hinges on, so it gets its own runnable verifier**.
> - **`reverse-telco-codebase-selftest.md`** (12:10) — 13-question self-test across 4 difficulty levels with hidden answer key; use *after* analysis to QA your work.
> - **`dag-reverse.md`** (17:20) — sister skill: notebook-first angle. Cross-references `reverse-telco-codebase` explicitly: "use this skill to build DAG structure and extract production parameters; fall back to `reverse-telco-codebase` when you need to verify date arithmetic internals." Codifies the breakthrough that notebooks contain orchestration metadata (`<component>` tags, `<airflow_upstream>`, `f_options`) that pre-empts source-code reading.

**Result.** Methodology now reusable beyond this audit — any future "how many weeks of clean data does pipeline X need?" question can run the same skill+verifier+self-test loop. The 13-question self-test in particular is a defence against confirmation bias on numeric findings (the kind that produced the original handcrafted `ds_estimate.csv` numbers in the first place).

## DCI-328 · MBF MHTT schema audit ships

**Why.** Tue's MHTT bundle was the first MBF-side delivery close to production shape. The Mar 24-27 audit had inferred the pipeline-side contract from `mbf_cs_airflow_202208` (`worker6_v4` + `consitution_v3.md`). Today validates the MBF side against that contract — file-to-table mapping, structural shape checks, gap report — so the bid-side ETL conversation can be specific about what MBF must change before the data is pipeline-ready.

**Glossary.** **MHTT** = Mobifone's billing/CDR file format (24 CSVs: subscriber, voice CDR, SMS CDR, recharge, balance, VAS, payment, ARPU, cell, VLR, handset, billing detail). **CDR** = Call Detail Record (one row per call/SMS event), vs the pipeline's expected hourly aggregate (one row per phone-day with `num_calls`, `total_duration`).

> ### 🛠 What I did
>
> - **Extracted `SAMPLE_MHTT/`** — 24 CSVs, 1000 rows each, encrypted ISDN.
> - **`audit_mhtt_schema.md`** — file-to-table mapping of all 24 MHTT files against the 21-table TS_Schema. Headline counts: **14 mapped pairs** (HIGH confidence); **7 MHTT files unmapped** (`TP_DTTT_TBTT`, `TP_DTTT_TBTS`, `TP_PROMO`, `TP_VITRI_PSC`, `TP_SMSC`, `TP_ICC_*`, `TP_ACTION_AUDIT`); **7 TS_Schema tables missing from MHTT** (deactivation, VAS extra, NBVIP, DND, MNP, exhaust, call_cell_id); **4 tables structural mismatch** (raw CDR delivered, pipeline expects hourly aggregates); **11 tables with column gaps**.
> - **`audit_mhtt_mapping.csv`** — machine-readable counterpart so the mapping can be filtered/sorted by the bid team without re-reading the markdown.
> - **`dag_reverse_clean_data_depth.md`** — walks the audit DAG backward from each `tc_*` clean table to the consumer function paths, with depth annotation. Lets the gap conversation be specific: "if MHTT can't deliver column X on table Y, jobs `<f1>`, `<f2>`, `<f3>` break, with a max downstream depth of Z."
> - **`audit_mhtt_derivation_supplement.md`** — derivations the pipeline does internally (OG+IC merge into `tc_call_histories_contact`, SMO+SMT merge into `tc_message_histories`, hourly-aggregate construction from raw CDR). The ETL between MBF and pipeline must cover these — the MBF side doesn't need to ship them pre-merged, it needs to ship the inputs.
> - **`dag_reverse_102A02_tc_files_needed.md`** — narrowed-scope drill-down: starting from Airflow ID `102A02` (the layer-2 anchor in `csv4_dev`), which `tc_*` files and columns are needed all the way down. Specific to the `csv4_dev/lib_notebook/102_csv4_layer2.ipynb` track that the customer conversation centers on.

**Result.** Bid-prep deliverable set in hand: schema audit + machine-readable mapping + reverse-DAG + derivation supplement + 102A02 drill-down. Top-level risks ranked: CRITICAL (raw CDR vs hourly aggregate), HIGH (direction-split files OG/IC + SMO/SMT vs combined-with-direction column, 7 missing tables), MEDIUM (encrypted ISDN must be decrypted in ETL, naming mismatches like `calling_isdn` vs `phone_number`). Sufficient input for the bid-side conversation about what ETL contract MBF needs to commit to.
