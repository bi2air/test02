## DCI-448 · Call-code diagnosis and fix-proposal automation

**Why.** This is the O4/KR1 automation anchor. The diagnosis flow must separate source evidence, runtime behavior, label disagreement, and taxonomy/config fix proposals. A bot or skill that jumps straight to a fix without evidence will create unreviewable changes.

> ### 🛠 What I did
> Built the diagnostic evidence layer around the sample pipelines. The Codex monitor validation keeps correctness separate from sample usability: correctness asks which label best fits the transcript, while signal review asks whether the transcript is reusable for a target or accepted label. The cross-family aggregator then joins what is already on disk into one per-cid decision row with a verdict, disagreement class, and follow-up routing. The PPIF/FDP case was preserved as an L2 boundary sample instead of being forced into a clean canonical bucket.

**Result.**

- `DCI-448` now has a concrete implementation direction: diagnosis first, fix proposal second.
- The current artifacts support traceable disagreement review: voter outputs, judge outputs, signal review, Codex/Opus cross-family comparison, and boundary-case routing.
