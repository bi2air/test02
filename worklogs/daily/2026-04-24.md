## AGI-9466 · SM-bot pilot — 18 conversations on the live Affirm bot

**Why.** AGI-9466 covers 11 ACs (NNP, DIM, GVT, DEC, DCP, HTT, EPT, ECH, TPM, FDP, TLM) on the Affirm 1P collection bot, targeting **≈100% L1 classifier accuracy**. Until today, evaluation ran only against synthetic seeds — useful for prompt iteration but never tested against the *real* SM bot. Without a real-bot baseline, AC failures couldn't be cleanly attributed (was it a classifier issue? seed/bot drift? a state-machine routing bug?). The pilot resolves that ambiguity.

> ### 🛠 What I did
> Stood up the AGI-9466 simulation pipeline (`docs/AGI9466_PIPELINE.md`, 399-line spec) on the zero server (`/home/binhnguyen2/explo/dirvenv/py310/bin/python`, workdir `/media/sdb/working/llm_voice/explo/binh2/postcall`). Ran `run_quality_loop_agi9466_all.py` for **18 conversations across 11 ACs at L1** — the easiest persona register, used as the classifier's positive-example baseline. Configuration: `gpt-4o T=0.7` debtor LLM, `gpt-4o T=0.0` reviewer LLM, rubric 0-5 with retry up to 3 attempts when score < 4. Validated against terminal-state and per-AC reviewer scores.

**Result.** Four production-aligning findings, captured in detail to `memory/project_agi9466_smbot_pilot_apr24.md`:
- **Entry state must be `pre_opening`**, not `inbound_opening` — `authenticate` is skipped on outbound calls. The synthetic harness was using the wrong start state, which is why synthetic transitions diverged from real bot state machines. Fixed.
- **`DEC` (deceased) triggers via `rpc=not`**, not `exc=death` as we'd assumed — AC-004 (deceased / PDJ) routing now uses the correct trigger. Direct production-routing fix.
- **`GVT` (Government shutdown) hits `max_turns` 100% of the time** — conversational flow doesn't terminate. Script-side handling bug isolated for the GVT path; flagged for fix.
- **Terminal-state match vs synthetic seeds: 16.7%** — synthetic seeds were materially easier than the real bot. Direct evidence that future evaluation must use real SM transcripts as ground truth, not synthetic-only.

**Credits.** Ha Viet LE (Product Director–AGI) — defined the original 11 AC seeds with mission_dict structure (primary_behavior, must_say_or_imply, must_not_say, distinct_from, etc.) that this pilot validates against.

## Real-transcript classifier baseline — 27 SM transcripts

**Why.** With the SM-bot pilot exposing the synthetic-vs-real gap, we needed a real-transcript classifier baseline to anchor future iterations against. Synthetic seeds had been the proxy until now; real transcripts are the ground truth.

> ### 🛠 What I did
> Pulled **27 real SM transcripts** spanning **L1/L2/L3 registers** across 9 AGI-9466 codes. (L1 = clear/easy persona; L2 = mixed/normal; L3 = stress test — broken flow, hostile, off-script.) Ran the dev-branch classifier locally; aggregated by register and by AC; surfaced the GVT systematic mis-classification path.

**Result.**
- **Overall: 11/27 (40.7%) exact match.** L1: 78% · L2: 44% · L3: 0%.
- **GVT: 1/9 across all levels** — systematic `shutdown → HCH` mis-classification. A single postprocess-condition path is responsible for every GVT failure across registers. Concrete and fixable in *one place* (not a per-case prompt-tuning problem).
- **Synthetic L1 was 36.4%; real L1 is 78%** — synthetic seeds were ~2× harder than real-bot conversations. Established the rule going forward: **always use real SM transcripts for classifier testing**, not synthetic L1.

**Links.**
- `memory/project_real_transcript_classifier_apr24.md` — captures the per-register split + the GVT systematic finding.

## Supporting infrastructure + bug fixes shipped today

**Why.** Three secondary issues surfaced in the morning that needed to be unblocked before the SM-bot pilot could scale across 11 ACs.

> ### 🛠 What I did
> **(1) `conv_id` Redis collision.** Reusing `conv_id` across runs caused new runs to read the *previous* run's final SM state from Redis (Redis is the cross-run shared store; without prefix the cache lookup is non-unique). Fixed by prefixing every `conversation_id` with a `run_token` — format `AGI9466_<run_token>_QA_<code>_L<n>_<rca>_a<N>`. Applied to `run_llm_debtor_agi9466.py`. *(`run_llm_debtor.py` and `run_dec_l1_rerun.py` still need the same patch — flagged for follow-up.)*
> **(2) L3 rubric incompatibility.** The L3 register (stress-test persona) fails to trigger AGI-9466 ACs in the outbound Affirm sim — **0/9 tasks passed across 27 attempts**. Concluded: L3 is a bot-side stress test, not a classifier-positive-example source. Future AGI-9466 evaluation should bound at L2 for positive examples; L3 stays as adversarial.
> **(3) `conv_examples` ablation.** Turned `conv_examples` off in `adverserial_tester_bk.py`; near-zero impact on strategy/persona output. Speaking-style is driven by the `instruction.yaml` template + `mission_dict` samples, not the example conversations. **New default: empty `conv_example` when scaling 11 ACs** — saves prompt budget without quality loss.

**Result.** Pilot output now interpretable; sweep can scale across 11 ACs cleanly. Prompt budget per AC trimmed via `conv_examples=empty` default.

## Synthetic RCA pool — 100 entries

**Why.** AC seeds without persona variation produce monotone conversations across runs. Scaling AGI-9466 across 11 ACs × multiple levels needed a reusable **RCA (Root Cause Analysis — the persona's reason for non-payment, e.g., medical bills, job loss)** pool with realistic variance.

> ### 🛠 What I did
> Wrote `src/vpbPostCall/src/adversarial/generate_synthetic_rca.py` (`seed=42`, reproducible). Output: `input/vpbPostCall/synthetic_rca_100.csv` — **100 entries, 24-key JSON RCAs**. Pairs with AGI-9466 AC seeds via `src/kompato/debtor/run_ac_with_rca.py`. Each RCA carries fields like reason category, financial details, family situation, hardship type — the demographics + circumstances that drive persona variance.

**Result.** 100-entry pool ready for the AGI-9466 sweep. Caveat documented honestly: **persona variance from RCA alone is *cosmetic*** — speaking style still tracks the L1/L2/L3 rubric, not the RCA. Real persona variance comes from coupling RCA + register. (Cosmetic missing field: `middle_name`.)

**Links.**

- `input/vpbPostCall/synthetic_rca_100.csv` — the 100-entry pool.
