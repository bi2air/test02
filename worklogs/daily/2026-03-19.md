## Telco Schema Matching Playbook v1+v2 + first audit pipeline run

**Why.** Wed staged two inputs (censored MBF sample, TS-side schema CSV) but no logic to bind them. **Round 2** under [DCI-328](https://trustingsocial1.atlassian.net/browse/DCI-328) needs a reviewable answer to send back to MBF — *for each TS-required column, which customer-delivered column maps to it, with what confidence, and what's the gap?* — and one that's reproducible across future telco customers, not one-off code. Today's deliverable is the playbook spec (how to do this for any telco) plus the first concrete run against MBF.

> ### 🛠 What I did
>
> - **Authored `matching_schema_telco_playbook.md` v1** — the Purpose / Inputs / Outputs / Step-by-step contract. v1 covers two outputs: (1) **Audit Report** (one row per customer CSV: column list, fill ratios at 10% / 50% / 90%, total rows, inferred type per column) and (2) **Schema Mapping** (one row per ETL column: matched customer file + column, confidence, samples from both sides). Encoded the type-inference regex table: `YYYY-MM-DD…` → datetime, 8-digit `20260314` → dateid, `[-+]?\d+(\.\d+)?` → number, ≥8-digit-heavy string → identifier, else string.
> - **Re-authored as v2 (`_v2.md`, ~64% larger)** — added two outputs the bid team flagged were missing: (3) **Gap Report** — ETL columns with no direct match, classified as `derivable` / `requestable` / `not_applicable`; (4) **Unmapped Customer Files** — customer files not assigned to any ETL file, flagged as potential new sources or irrelevant extras. Restructured the doc into three Phases: Discovery (parse both sides), Matching (column-level confidence scoring with primary + secondary candidates), Reporting (the four CSVs).
> - **Built `schema_check01.ipynb`** as the driver notebook — loads the ETL Schema CSV, the customer folder, and the staging JSONs (`mbf_sample_20260318.json` for column lists, `ts_etl_mbf_schema.json` for the structured TS schema). The notebook is what `enrich_csv.py` / `reverse_mapping*.py` plug into for the first audit run.
> - **Shipped the first production run** of the v2 pipeline against the censored MBF sample, producing the four-file output set tagged `_20260319`: `audit_report`, `schema_mapping`, `gap_report`, `unmapped_files`. The small `unmapped_files` size (~3-4 file names) and small `gap_report` are positive Round-2 signals — almost every customer file mapped to something, and few TS columns are completely uncovered.

**Result.** End-to-end pipeline shipped in one day, from "two staged inputs" to "four reproducible CSVs". The playbook v2 becomes the reusable spec for later telco customers under the [DCI-321 Telco Score Maintenance](https://trustingsocial1.atlassian.net/browse/DCI-321) parent; the four `_20260319` outputs become the baseline that Fri's encrypted-sample re-run is compared against.
